# v0.3 Phase 3: Operation Learning Design

## Goal
Learn when to perform which memory operation: ADD, UPDATE, DELETE, NOOP.

## The Problem
Currently, Association always ADDs new memories when they meet certain criteria
(threshold importance, novel keywords). But this is naive:

- "Sophie lives in London" then "Sophie lives in the UK" → should UPDATE, not ADD
- "I'm working on X" then "I finished X" → should UPDATE or DELETE
- "the weather is nice" → probably NOOP (too ephemeral)

## Memory-R1 Approach
Memory-R1 uses RL to learn operations. The Memory Manager agent takes:
- Current state (extracted info + existing memories)
- Outputs: operation ∈ {ADD, UPDATE, DELETE, NOOP} + content

## MVP Approach for Association

We can't do full RL without training infrastructure, but we can:

### Heuristic-Based Operation Decision

1. **NOOP**: Skip storage if:
   - Content is too short (< 20 chars)
   - No interesting keywords
   - Very similar to recently surfaced memory (relevance > 0.9)
   - Content is purely conversational ("ok", "thanks", "yeah")

2. **UPDATE**: Merge if:
   - High similarity to existing memory (> 0.8 keyword overlap)
   - Same topic/entity but new information
   - Contradicts or refines existing memory

3. **DELETE**: Remove if:
   - Memory is stale (not accessed in 30+ days)
   - Memory consistently surfaces but never used (feedback tracking)
   - Contradicted by new, more reliable information

4. **ADD**: Default if none of the above

### Implementation

```typescript
interface OperationDecision {
  operation: 'ADD' | 'UPDATE' | 'DELETE' | 'NOOP';
  targetId?: string; // For UPDATE/DELETE
  confidence: number; // How confident in this decision
  reason: string; // Why this operation
}

function decideOperation(
  content: string,
  keywords: string[],
  surfaced: AssociatedMemory[],
  feedback: MemoryPerformance[]
): OperationDecision {
  // 1. Check NOOP conditions
  if (isEphemeral(content) || isRedundant(content, surfaced)) {
    return { operation: 'NOOP', confidence: 0.8, reason: 'ephemeral or redundant' };
  }

  // 2. Check UPDATE conditions
  const similarMemory = findSimilarMemory(keywords, surfaced);
  if (similarMemory && shouldMerge(content, similarMemory)) {
    return {
      operation: 'UPDATE',
      targetId: similarMemory.memory.id,
      confidence: 0.7,
      reason: 'similar topic with new info'
    };
  }

  // 3. Check DELETE conditions (for existing memories)
  // This is called during maintenance, not during process()

  // 4. Default to ADD
  return { operation: 'ADD', confidence: 0.6, reason: 'new unique content' };
}
```

### UPDATE Logic

UPDATE is the hardest. How do we know when to merge vs add separately?

Signals for UPDATE:
- Same named entities (person, project, place)
- Topic overlap > 80%
- Temporal sequence (update follows original)
- Refinement keywords: "also", "actually", "update", "now"

Example:
- Existing: "Sophie builds OriginOS"
- New: "Sophie also works on OSL language"
- Decision: UPDATE → "Sophie builds OriginOS and works on OSL language"

### DELETE Logic

DELETE should be conservative. Only delete when:
- Memory is stale (not accessed in 30+ days) AND
- Has low success rate (< 20% used when surfaced) OR
- Is contradicted by newer, verified information

DELETE candidates are found via `findPrunableMemories()`.

### Training Data (Future)

If we had training data, we could learn:
- Which content patterns → NOOP
- Which similarities → UPDATE vs ADD
- Which feedback patterns → DELETE

For now, heuristics + feedback signals provide a reasonable baseline.
