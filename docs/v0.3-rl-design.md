# Association v0.3: RL-Driven Memory Operations

## Background

Memory-R1 (arXiv:2508.19828) showed that RL can optimize memory management:
- Two-agent architecture: Manager (operations) + Answer (retrieval)
- Operations: ADD, UPDATE, DELETE, NOOP
- Outcome-driven learning via PPO + GRPO
- Outperforms hand-crafted systems (LOCOMO, LangMem, A-Mem, Mem0, Zep)

## Current State (v0.2)

Association uses hand-crafted heuristics:
- **Storage**: keyword extraction + fuzzy matching
- **Scoring**: fixed weights (keyword 0.4, recency 0.2, importance 0.3, novelty 0.1)
- **Deletion**: none (only compression)
- **Retrieval**: triggered by keyword overlap above threshold

## v0.3 Goals

1. Learn which memories are actually useful (not just keyword-match)
2. Adapt scoring weights based on retrieval success
3. Implement intelligent pruning (DELETE operation)
4. Track memory usefulness over time

## Design

### Feedback Loop

The key insight from Memory-R1: you need outcome feedback to learn.

For Association, "success" means:
- Surfaced memory was referenced in the response
- Surfaced memory led to correct/good answer
- Memory was NOT surfaced but should have been (negative signal)

Implementation:
```typescript
interface MemoryFeedback {
  memoryId: string;
  surfaced: boolean;      // Was this memory surfaced?
  used: boolean;          // Was it referenced in response?
  helpful: boolean | null; // Did it lead to good outcome? (null = unknown)
  timestamp: Date;
}
```

### Reward Signal

Simple reward for MVP:
- +1 if memory was surfaced AND used
- -1 if memory was surfaced but NOT used (noise)
- +0.5 if memory was NOT surfaced but would have been helpful (miss)

The "would have been helpful" signal requires the agent to notice when it needed memory but didn't get it. This is harder to track.

### Adaptive Scoring Weights

Start with a simple approach: track per-weight performance.

```typescript
interface WeightPerformance {
  weight: 'recency' | 'importance' | 'novelty' | 'keyword';
  successRate: number;  // How often memories surfaced by this weight were used
  sampleCount: number;  // How many observations
}
```

Adjust weights periodically based on success rates:
- If recency-weighted memories are rarely used, decrease recencyWeight
- If importance-weighted memories are often used, increase importanceWeight

### Memory Operations (RL-style)

Instead of always storing, consider:
- **NOOP**: Don't store this message (low importance)
- **ADD**: Store as new memory
- **UPDATE**: Merge with existing memory (duplicate/similar topic)
- **DELETE**: Remove stale/incorrect memory

For MVP, focus on DELETE learning:
- Track memory "staleness" signals:
  - Not accessed in N days
  - Surfaces but never used
  - Contradicted by newer information

### Implementation Phases

**Phase 1: Feedback Collection** (this heartbeat)
- Add `recordFeedback()` method
- Track surfaced/used/not-used
- Store feedback in memory metadata

**Phase 2: Weight Adaptation**
- Analyze feedback to adjust weights
- A/B test different weight combinations
- Measure impact on retrieval quality

**Phase 3: Operation Learning**
- Learn when to NOOP (skip storage)
- Learn when to DELETE (prune stale)
- Learn when to UPDATE (merge duplicates)

## Open Questions

1. **Feedback source**: How does the agent signal "I used this memory"?
   - Option A: Explicit API call `association.markUsed(memoryId)`
   - Option B: Parse agent response for memory references
   - Option C: Ask agent to rate surfaced memories

2. **Negative signal**: How do we know a memory should NOT have been surfaced?
   - No clear signal unless agent explicitly says "that's not relevant"

3. **Privacy**: Feedback data could reveal conversation contents
   - Store only metadata (memory IDs, timestamps, success flags)
   - Don't store conversation content in feedback

## MVP for v0.3

Smallest useful RL feature:
1. Track memory access outcomes
2. Adjust scoring weights weekly based on success rates
3. Flag memories for deletion if consistently unused

This is a step toward self-improving memory, not full RL.
